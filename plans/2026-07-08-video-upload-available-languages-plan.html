Repair Video Upload Available Languages Sync

Production has playable video variants for the requested language uploads, but several are missing from Video.availableLanguages. That makes direct playback work while discovery/listing endpoints lag or miss the language.

Created 2026-07-08 Scope: api-media upload processing Primary video: 1_jf-0-0 Review mode: one step at a time
Observed Direct playback is live

The direct Arclight component-language route returns HLS for Mupun, Yao Baiku, Langilan Manobo, Talaandig Manobo, and Ama.

Observed Discovery is stale

availableLanguages includes Yao Baiku and Ama, but not Mupun, Langilan Manobo, or Talaandig Manobo.

Likely Cause Upload worker bypasses side effects

Direct Prisma writes create/update variants without running the same language-maintenance code used by the GraphQL mutation.

Recommendation

Fix the upload-processing code path so every successful published variant creation/update also updates availableLanguages, parent collections, Algolia, and cache. Then run a one-time production repair to recompute stale availableLanguages from current published variants.

Decision point: We should decide whether to patch only the current upload worker, or refactor both upload-worker paths to call one shared createOrUpdateVideoVariant implementation with side effects included.

Code Findings

Known-good mutation path

apis/api-media/src/schema/videoVariant/videoVariant.ts calls addLanguageToVideo and updateParentCollectionLanguages after creating a published variant.

Upload worker path

apis/api-media/src/workers/processVideoUploads/service/service.ts creates or updates VideoVariant directly and does not run those side effects.

Upload service helper path

apis/api-media/src/schema/videoVariantUpload/service.ts has another direct create/update helper with the same missing language sync concern.

Repair helper exists

updateVideoAvailableLanguages can recompute the field from currently published variants and child video languages.

Walkthrough Steps

1

Confirm desired behavior

Agree that any successfully published upload-created variant should immediately appear in language discovery, parent collection/feature-film language lists, and Algolia search.

  • Confirm whether unpublished uploads should be excluded.
  • Confirm whether existing variant updates should also refresh the aggregate list.
2

Choose implementation shape

Prefer one shared helper for upload-created variants so both the worker and resume/service path cannot drift again.

  • Option A: add side-effect calls after each direct create/update.
  • Option B: centralize create/update and side effects in one helper.
  • Recommendation: Option B if the diff stays small; Option A if release speed dominates.
3

Patch upload variant finalization

After the variant is created or updated and is published, call addLanguageToVideo(videoId, languageId), updateParentCollectionLanguages(videoId), updateVideoInAlgolia(videoId), and reset relevant caches.

  • Keep variant creation atomic enough that failures are visible.
  • Send Slack failure notifications if language sync fails after upload finalization.
  • Avoid breaking successful playback if a non-critical cache/index refresh fails.
4

Add regression tests

Tests should prove that upload finalization updates the variant and the aggregate language fields.

  • Worker test: ready Mux asset creates/updates variant and adds language to the video.
  • Parent test: child video language update refreshes parent feature-film/collection language list.
  • Negative test: unpublished variant does not add to availableLanguages.
5

Create one-time repair

Run a bounded production repair for affected videos, starting with 1_jf-0-0, using the existing recompute helper or a tiny script that calculates languages from published variants.

  • Dry-run output should show before/after language IDs.
  • Apply run should update DB, Algolia, and cache.
  • Capture the exact affected IDs in the deploy notes.
6

Validate production

Recheck both direct playback and discovery paths after deploy and repair.

  • /v2/media-components/1_jf-0-0/languages/{languageId} returns 200 with HLS.
  • /v2/media-components/1_jf-0-0?expand=languageIds includes all five IDs.
  • /v2/media-components?languageIds={languageId}&subTypes=featureFilm returns 1_jf-0-0.

Risks And Guardrails

  • A direct DB repair without cache/index refresh can leave Arclight or Algolia stale, so repair should include cache reset and Algolia update.
  • If the worker retries after partial success, the language update must be idempotent. addLanguageToVideo already handles duplicate prevention.
  • Exact version confirmation is publisher/DB-gated; REST proves playback, while DB or authorized GraphQL is needed to prove the stored version values.

Open Questions

  1. Should the repair target only 1_jf-0-0, or recompute all videos with upload-created variants from the last few days?
  2. Should missing download URLs for Yao Baiku block the rollout, or be tracked as a separate Mux download-processing issue?
  3. Do we want upload finalization to update Algolia synchronously, or queue that update separately to reduce worker latency?

Definition Of Done

  • New upload-created published variants update availableLanguages immediately.
  • Parent collection/feature-film language lists update after child variant finalization.
  • Regression tests cover the upload-worker path.
  • Production repair makes all five requested languages discoverable.
  • Post-repair checks confirm direct playback and listing endpoints agree.
