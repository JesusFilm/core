# Context Map

This is an Nx monorepo with multiple bounded contexts — each `app`, `api`, and shared `lib` is (or will become) its own context with its own `CONTEXT.md`. This map lists the contexts that have been modeled so far and how they relate. Contexts are added lazily as `/domain-modeling` documents them; an entry appears here only once its `CONTEXT.md` exists.

## Contexts

- [Journeys](./apis/api-journeys/CONTEXT.md) — the authoring and delivery of interactive, block-based journeys (evangelism funnels): journeys, their block/action content, owning teams, templates and the gallery, and distribution (custom domains, QR codes).
- [Journey Analytics](./apis/api-journeys/CONTEXT-analytics.md) — the audience-and-measurement side of journeys: visitors, the event stream, conversion goals (Plausible), and the outflow of captured responses to external systems. Shares the `api-journeys` deployable and database with Journeys.
- [Plausible Provisioning](./apis/api-analytics/CONTEXT.md) — the measurement-provisioning subgraph (deployable `api-analytics`): a thin GraphQL layer over Plausible Analytics' own database that provisions the Sites, Goals, and Shared Links traffic is reported against. Authenticates a Plausible identity (API key), owns no schema of its own.
- [Media](./apis/api-media/CONTEXT.md) — the catalog of films and their language-specific renditions, subtitles, assets, and metadata.
- [Users](./apis/api-users/CONTEXT.md) — user identity as a read-through projection of Firebase Auth: profile, email verification, admin status, impersonation, and account deletion.
- [Languages](./apis/api-languages/CONTEXT.md) — the reference catalog of languages, the countries and continents they are spoken in, and the localized names of all three.
- [Arclight](./apps/arclight/CONTEXT.md) — the legacy-compatibility REST façade (`api.arclight.org` vocabulary): Media Components, Media Languages/Countries, taxonomies, and redirect/short-link delivery for external API-key consumers. Owns no entities; a pure projection of the media and languages contexts.
- [Gateway](./apis/api-gateway/CONTEXT.md) — the federation edge: composes the subgraphs into one supergraph, verifies caller identity (Firebase JWT) once at the door, and signs requests to subgraphs. Owns no domain entities.
- [Journeys Viewer](./apps/journeys/CONTEXT.md) — the audience-facing delivery surface (NextSteps viewer): plays back published journeys to anonymous visitors — playback/navigation state (Conductor, block history), delivery modes (root/custom domain, embed, website mode), visitor session creation, and the AI assistant. Owns no content entities.
- [Journeys Admin](./apps/journeys-admin/CONTEXT.md) — the creator-facing authoring surface: the visual Editor (JourneyFlow/Canvas/Drawer spatial model, undoable Commands, Analytics Mode), journey lifecycle management (Active/Archived/Trash), template adoption and Publisher curation, sharing flows, and the Reports/Visitors measurement surfaces. Owns no entities; the authoring experience over the Journeys and Journey Analytics contexts.
- [Player](./apps/player/CONTEXT.md) — the audience-facing shared-playlist viewer (`player.jesusfilm.org`): plays a media-context Playlist shared by its owner (slug-addressed), and hands viewers off to the native mobile apps (install banners, deep links). Owns no entities.
- [Resources](./apps/resources/CONTEXT.md) — the public `jesusfilm.org` content surface: a next-gen rebuild of the Watch product (the video library in 2,000+ languages), plus the Resources section (WordPress-authored ministry assets) and a Journeys template gallery. Owns no entities.
- [Short Links](./apps/short-links/CONTEXT.md) — the public redirect surface: serves every Short Link Domain from one deployment, resolves hostname + path to a media-owned ShortLink via the gateway, and redirects to its Destination. Owns no entities.
- [Video Importer](./apps/video-importer/CONTEXT.md) — the operator-run batch ingest CLI for the media catalog: classifies local files by filename convention and pushes variants (via Mux), subtitles, and audio previews into the media context. Owns no entities; its domain is the filename contract and the run.
- [Watch Modern](./apps/watch-modern/CONTEXT.md) — an early-stage successor shell for the `/watch` surface, mounted at `/watch` via proxy routing to coexist with the legacy app; today a scaffold (landing page, locale handling, legacy `watch.html` redirect shim) with no video catalog implemented. Owns no entities.
- [Watch (legacy)](./apps/watch/CONTEXT.md) — the current production `jesusfilm.org/watch` surface: the video library in 2,000+ languages (Algolia-browsed, video.js playback), the `.html` Watch Path URL scheme, viewer language preferences, and the Easter campaign pages. Owns no entities; being succeeded by Resources.
- [Videos Admin](./apps/videos-admin/CONTEXT.md) — the publisher-facing curation surface for the media and languages catalogs: Video publishing workflow (Draft/Publish All/Locked/Restricted), variant ("Audio Language") uploads via R2→Mux, editions/subtitles/images, and Language reference-data editing (names, codes, country links). Owns no entities.
- [E2E Testing](./apps/CONTEXT-e2e.md) — the end-to-end verification surface: the eleven Playwright suites (`apps/*-e2e`) and their shared vocabulary of Deployment URLs, the Daily E2E run, the Playwright User pool, Monitors, and arclight's Contract Specs. Owns no product entities; one context spanning all `apps/*-e2e` directories.
- [Journeys UI (shared kernel)](./libs/journeys/ui/CONTEXT.md) — the shared client-side vocabulary of journey rendering and editing (`libs/journeys/ui`): the Tree Block model and Block Renderer, playback state (Block History, Action Handling, Event Mutations), editor state (Editor Provider, Commands), render modes, and the shared Template Gallery/search components. Owns no entities; the kernel consumed by Journeys Viewer, Journeys Admin, and Resources.
- [Prisma (persistence kernel)](./libs/prisma/CONTEXT.md) — the shared persistence layer (`libs/prisma`): one Prisma schema per context-owned Logical Database in the `jfp-core` Cluster (journeys, media, languages, users, plus the introspected Plausible database), and the client/migration/seed conventions around them. Not a domain; each subdirectory is another context's storage.
- [Infrastructure](./infrastructure/CONTEXT.md) — the platform substrate: the OpenTofu/Terraform estate (AWS `us-east-2`) defining the `prod`/`stage` Environments, the shared network/compute Substrate, each ECS Service's runtime, the shared Core Database and Redis, and the Atlantis PR-driven change workflow. Owns no product entities.
- [Shared AI (LLM kernel)](./libs/shared/ai/CONTEXT.md) — the shared vocabulary of calling LLM providers (`libs/shared/ai`): Fallback Sessions over Model Chains (Gemini and OpenRouter flavors), Prompt Hardening (guillemets + the Hardening Prompt), the Journey Simple schema (the AI-facing journey shape), and the Image Description helper. Owns no entities; consumed today only by `api-journeys`' AI features.
- [Yoga (subgraph runtime kernel)](./libs/yoga/CONTEXT.md) — the shared server-side kernel of the GraphQL Yoga subgraph APIs (`libs/yoga`): caller-identity parsing (Payload User), the Interop Token service-trust mechanism, the Gateway Callback Client, Leader Lock worker election, the NextSteps-flavored Email Kit, and symmetric-encryption helpers. Owns no entities; consumed by all six APIs.
- [Dev Hosts (dev-trust kernel)](./libs/shared/dev-hosts/CONTEXT.md) — the shared vocabulary of trusting a developer's own hostname (`libs/shared/dev-hosts`): the secret-provisioned Dev Hosts Registry, the fail-closed membership check, and the per-surface Relaxations gated on it. Owns no entities; consumed by frontend surfaces (journeys, journeys-admin) and intended for the gateway.
- [ESLint (lint-policy kernel)](./libs/shared/eslint/CONTEXT.md) — the monorepo's shared lint policy (`libs/shared/eslint`): the Common Base of house rules and the per-workspace-type Flavors (`next`, `api`, `yogaWithReactEmail`, `prisma`, `e2e`) composed into every workspace's `eslint.config.mjs`. Owns no entities; pure engineering policy.
- [GQL (supergraph typing kernel)](./libs/shared/gql/CONTEXT.md) — the shared vocabulary of type-safe GraphQL authoring (`libs/shared/gql`): gql.tada bound to the gateway's composed Supergraph Schema, giving every client a `graphql()` tag whose Typed Operations infer result/variable types with no per-operation codegen. Owns no entities; types-only.
- [Shared UI Dynamic (code-splitting façade)](./libs/shared/ui-dynamic/CONTEXT.md) — a thin façade over the Shared UI kernel (`libs/shared/ui-dynamic`): per-component Dynamic Entries (today only `Dialog`) that let journeys-admin lazy-load Shared UI components as named webpack chunks. Owns no components; not a context so much as a build-boundary shim.
- [Storybook (component-catalog shell)](./libs/shared/storybook/CONTEXT.md) — the monorepo-wide component catalog (`libs/shared/storybook`): a source-less Shell Project running the single root `.storybook/` config, aggregating Registered Projects' stories (UI surfaces and API email templates) into one catalog and one Visual Regression build (Chromatic, currently dormant). Owns no components.
- [Shared UI (design-system kernel)](./libs/shared/ui/CONTEXT.md) — the shared client-side design vocabulary (`libs/shared/ui`): the MUI Theme catalog (`base`/`journeyUi`/`journeysAdmin`/`website` × light/dark), the ~244-icon house Icon Set, Feature Flag plumbing, the Storybook Config Kit, and cross-app components. Owns no entities; consumed by every Next.js surface and `libs/journeys/ui`.
- [Shared UI Modern (Tailwind design kernel)](./libs/shared/ui-modern/CONTEXT.md) — the Tailwind-era sibling of the Shared UI kernel (`libs/shared/ui-modern`): vendored shadcn/ui components (added via the Add Script), the HSL-variable Design Tokens, the `cn` class-merging helper, and the Extended Component pattern for house variants. Owns no entities; consumed today only by Watch (legacy).
- [JF Proxy (edge routing worker)](./workers/jf-proxy/CONTEXT.md) — the Cloudflare Worker on the `jesusfilm.org` zone (`workers/jf-proxy`): Route Patterns claiming `/watch*`, `/journeys*`, `/resources`, `/bin/*`, `/api/*`… and proxying `/watch` to the Watch destination and other claimed paths to the Resources destination, with a 404/500 Error Fallback and mobile App Association Files. Owns no entities; pure edge plumbing.
- [Langfuse Export (chat-insights tooling)](./tools/langfuse-export/CONTEXT.md) — the engineer-run export pipeline (`tools/langfuse-export`): pulls Apologist chat traces from Langfuse, PII-scrubs them into Sanitised Conversations, and packages the lossless Dataset plus offline viewer into a shareable Explorer Bundle. Owns no entities; a read-only projection of the AI chat's observability data.
- [Load Test (chat load-generation tooling)](./tools/load-test/CONTEXT.md) — the engineer-run k6 load generator (`tools/load-test`): YAML Scenarios driving a chat Target, per-Run Results committed as evidence, and the Firewall Trip exercise against the Vercel rate-limit rule (stage only). Owns no entities.
- [Scripts (CI/ops glue)](./tools/scripts/CONTEXT.md) — the repo-operations scripts (`tools/scripts`): Affected Projects → Deployable Services feeding CI matrices, the Stage Reset over "on stage" PRs, Preview Deployment Comments, and one-off ops utilities. Owns no entities; the CI/deployment pipeline's vocabulary.
- [Docs](./apps/docs/CONTEXT.md) — the developer handbook site (`docs.core.jesusfilm.org`, Docusaurus): prose about the Core Development Kit, engineering practices, and the team. Owns no entities; a documentation projection of all other contexts.

_Not yet documented (add as they are modeled): the remaining shared `libs/*`._

_Tooling contexts (`tools/*`) are engineer-facing and own no product entities; they appear above alongside the kernels._

## Relationships

- **Media → Languages**: Media references languages only by `languageId` (primary language, per-Variant language, per-text language). The Language entities themselves are owned by the languages context, not by Media.
- **Media → Users**: Media references people only by user id (a Playlist's owner, a User Media Profile, a Media Role). User identity is owned by the users context; Media stores only its media-facing view of a user.
- **Media → consuming Platforms**: The `journeys`, `watch`, and `arclight` surfaces consume Media as a `Platform`; a Video can restrict which Platforms may view or download it.
- **Users ↔ Firebase Auth**: Firebase is the external source of truth for authentication; Users holds a read-through projection (a `User` row per Firebase Auth record), created lazily on first sight. The two can diverge and are reconciled during lookup and deletion.
- **Users → other contexts (federation)**: Users owns the `User` entity and other contexts reference it by id. Note the identity asymmetry — the federation/inbound key is the **Firebase UID** (`userId`), but the `id` field Users _returns_ is the **User record id** (DB UUID). See [Users CONTEXT.md](./apis/api-users/CONTEXT.md).
- **Languages ← Media (has-videos flag)**: Media is the catalog of videos and references languages by `languageId`; Languages stores a denormalized `hasVideos` flag per Language (set upstream, not written by either API's resolvers) that gates public visibility. Languages owns the language/country reference data; Media owns the videos.
- **Languages → other contexts (federation)**: Languages owns the `Language`, `Country`, and `CountryLanguage` entities and federates them by id; other contexts reference a language only by `languageId`. The `AuthenticatedUser` type is extended here to expose a user's `languageUserRoles` (the `publisher` role).
- **Languages source of truth (outward dump)**: Production `api-languages` is authoritative; its database is dumped to object storage and that dump seeds lower and local environments via a full restore. Publisher edits in production persist; edits made downstream are overwritten on the next import.
- **Journeys ↔ Journey Analytics**: Two bounded contexts inside one deployable (`api-journeys`), sharing the journeys database. Journeys authors the **Journey**, **Block**, and **Team**; Journey Analytics references them by id and records the **Visitor**/**Event** activity against them. They share the **Message Platform** vocabulary but hold different notions of `userId` — a _creator_ (Firebase UID) in Journeys, an _anonymous audience device/session_ in Analytics (see the `userId` trap in [Journey Analytics CONTEXT.md](./apis/api-journeys/CONTEXT-analytics.md)).
- **Journey Analytics → external systems (data outflow)**: Captured audience responses leave via team-level **Integrations** — exported into Google Sheets (`GoogleSheetsSync`) and pushed to Growth Spaces routes for follow-up — while aggregate metrics are measured through Plausible sites scoped by `plausibleToken`.
- **Journeys → Plausible Provisioning (site provisioning)**: `api-journeys`' plausible worker calls `api-analytics`' single `siteCreate` mutation through the gateway to provision one Plausible **Site** per team, journey, and template — using a synthetic `domain` (`api-journeys-<team|journey|template>-<id>`) as the site key and storing the returned Shared Link `slug`/`plausibleToken` back on the journey/team. Plausible Provisioning owns the Plausible Site; Journeys holds only the token and site key.
- **Plausible Provisioning ↔ Plausible (introspected external DB)**: `api-analytics` does not own its data model — it reads and writes Plausible Analytics' own Postgres tables directly (schema pulled via `nx prisma-introspect prisma-analytics`, no migrations). Its identity space is Plausible's `users` table, authenticated by a Plausible **API key** — entirely separate from the Users context / Firebase.
- **Journeys → Users**: Journeys never stores identity. Team memberships (`UserTeam`), journey access (`UserJourney`), invites, and the `Host`/creator surfaces all reference a creator by **Firebase UID** only; the `User` is resolved by federation from the users context. Note the **Visitor `userId` is a different identity space entirely** — an anonymous audience device/session token, not a `User` — and must never be joined to a creator userId. See [Journeys CONTEXT.md](./apis/api-journeys/CONTEXT.md).
- **Journeys → Users (deletion saga)**: Account deletion spans contexts. The journeys context computes the journeys/teams/memberships to remove and hands the id lists to Users' `userDeleteConfirm`, which deletes the identity and records them in its User Delete Audit Log. Users owns identity; journeys owns membership. Shared kernel: both halves narrate progress with the same **Log Entry** type (`libs/shared/user-delete` — a type-only lib, not a context of its own); this ephemeral operator-facing narration is distinct from the durable Audit Log.
- **Journeys → Languages**: A Journey has a `languageId`, and blocks reference languages by id (`videoVariantLanguageId`, `subtitleLanguageId`, per-text `languageId`). The `Language` entities are owned by the languages context; Journeys stores only the ids.
- **Journeys → Media**: A `VideoBlock` references a video by id and `VideoBlockSource` (`internal` = a Media video, plus `mux`/`youTube`/`cloudflare`); `TemplateGalleryPageMedia` references a Mux video by id with denormalized metadata. The `mediaVideo` module federates video details from the media context; Journeys owns none of the media.
- **Journeys → Short Links (gateway federation)**: A `QrCode` carries a `shortLinkId` that resolves to a `ShortLink` entity owned by the **media** subgraph (`api-media` serves the `shortLink*` fields) and stitched at the gateway; Journeys stores only the id and builds the redirect URL.
- **Arclight → Media**: Arclight's **Media Component** _is_ the media context's **Video** (renamed), and **Media Component Language** is a **Variant** (`refId` = variant id); Sub Type is the video's `label`, while Component Type is derived from playability, not stored. Arclight reads via the gateway and directly from the media database; it owns nothing. Keyword Short Links reference the media-owned **ShortLink** by path.
- **Arclight → Languages**: Media Languages/Countries are projections of the languages context's **Language**/**Country**, referenced by numeric `languageId`/`countryId`. Consumers speak BCP-47 **Metadata Language Tags** for descriptive text; Arclight resolves tags → language ids internally, falling back to English (id `529`).
- **Arclight → external consumers**: Third-party apps and partners identified by legacy **API Key** query strings; the key selects a default **Platform** (ios/android/web) and can fork behavior for specific distribution partners. Media/stream delivery redirects out to Brightcove; search is served from Algolia indexes of media + languages data.
- **Journeys Viewer → Journeys (content consumption)**: The viewer fetches a published Journey by **Slug** (scoped by incoming hostname for **Custom Domains**) through the gateway and renders its Block tree; drafts are excluded. It owns none of the content model — Journey/Block/Step/Card/Action vocabulary is the Journeys context's. Shared rendering primitives live in `libs/journeys/ui`, consumed by both the viewer and `journeys-admin`.
- **Journeys Viewer → Journey Analytics (event origination)**: The viewer is where the **Visitor** is minted (anonymous Firebase sign-in on the device, enriched with country/referrer) and where the event stream originates — journey-view and step-navigation events are observed on the client and reported to Journey Analytics (and mirrored to GTM/Plausible). Playback state itself (block history, active step) never leaves the device except as these events.
- **Journeys Admin → Journeys (authoring)**: The admin is where the Journeys context's content is authored — Journey/Block/Step/Card, Team, Template, roles, invites, Custom Domains, and QR Codes are all upstream vocabulary; the admin owns only the Editor's spatial model (Slides, JourneyFlow, Canvas, Drawer, Properties), the lifecycle actions (Archive/Trash/Delete Forever), and the adoption flows (Use This Template, Quick-Start Customization). Terminology trap: the admin's gallery **Collection** is the upstream **Template Gallery Page**, _not_ the upstream **Journey Collection**. Editor state vocabulary (`EditorProvider`, Commands, selected step/block) lives in the shared `libs/journeys/ui`.
- **Journeys Admin → Journey Analytics (measurement consumption)**: The admin reads what the viewer wrote — Visitors, Events, and Plausible stats — via three surfaces: Analytics Mode overlaying the JourneyFlow, the Reports area (Power BI for aggregate BI, the Plausible Embed Dashboard per journey), and the Visitor Timeline/Status triage views. It also configures the data outflow: team Integrations (Growth Spaces routes, Google Sheets Sync) are edited here, executed upstream.
- **Player → Media (playlist consumption)**: Player fetches the media context's **Playlist** by slug through the gateway (client name `player`) and renders its Items' **Variants** — HLS stream, localized title/description/study questions, thumbnail. It owns none of the content: Playlist/Item/Variant/Video are media vocabulary; the **Language** shown per item is federated from the languages context. A Variant without an HLS stream is an Unplayable Item on this surface.
- **Player → native mobile apps (handoff)**: Playlist URLs double as deep links — App Association files (`.well-known`) and the iOS Smart App Banner open the same Playlist in the native apps, and the Install Banner steers new viewers to the stores. Playback telemetry goes to Mux only; unlike the journeys viewer, Player mints no Visitor and emits no Journey Analytics events.
- **Resources → Media (watch consumption)**: The `/watch` surface renders the media context's **Video**/**Variant** catalog — Container Videos (label `collection`/`series`/`featureFilm`) as browse pages, Content Videos as playback pages — fetched via the gateway and searched via Algolia indexes of media data. All video vocabulary is upstream; the app owns only the Watch Path URL scheme and the legacy `watch.html` redirect shim.
- **Resources → Languages**: Watch Paths embed language slugs; the audio/subtitle preference UI and locale detection are built on the languages context's **Language** catalog (referenced by `languageId`, English `529` default). The app caches the catalog but owns none of it.
- **Resources → Journeys (gallery consumption)**: The `/journeys` section renders the journeys context's **Template Gallery** and template views via the shared `libs/journeys/ui` — the same upstream vocabulary as `journeys-admin`; nothing journey-shaped lives here.
- **Resources → WordPress (resource content)**: The **Resource** entities on `/resources` are authored in the external `jesusfilm.org` WordPress and reach the app only through Algolia indexes; no API context in this repo owns them.
- **Resources ↔ Watch (succession)**: `apps/resources` is the next-generation rebuild of the legacy `apps/watch` surface — same `/watch` product, same media vocabulary. Terminology trap: "Collection" means three different things across these surfaces (media Video Label, the Easter campaign page, the journeys Journey Collection) — see [Resources CONTEXT.md](./apps/resources/CONTEXT.md).
- **Short Links → Media (resolution)**: The redirect surface is a read-only gateway client of the media context's `shortLinkByPath` — **ShortLink**, **ShortLinkDomain**, and the blocklist are all media-owned; the app reads only the Destination and issues the redirect. The video-delivery fields on ShortLink (Brightcove id, redirect type, bitrate) are consumed by Arclight, never by this surface.
- **Short Links ↔ Arclight (complementary redirect surfaces)**: Requests to the `arc.gt` domains are handed off by Short Links to the Arclight API, whose Keyword Short Link route resolves the same media-owned entities (with Brightcove video redirects); every other Short Link Domain is resolved by Short Links itself. Same entity, two surfaces, split by domain.
- **Short Links ← Journeys (QR traffic)**: Journeys' **QrCode** stores a `shortLinkId` and builds its scan URL from the media-owned ShortLink; scans therefore arrive at this surface (or Arclight, per the domain split) to be redirected to the journey.
- **Video Importer → Media (ingest)**: The importer is a write-side gateway client of the media context — it uploads bytes directly to R2, then calls media mutations to register assets, enqueue Mux encoding (async — "imported" ≠ "playable"), and upsert subtitles/audio previews. Pre-existence rule: it never creates a **Video**, **Edition**, or **Language**; those must already exist in the catalog. Terminology trap: a Burned-In Filename keys the created Variant by the on-screen subtitle language, not the audio language — see [Video Importer CONTEXT.md](./apps/video-importer/CONTEXT.md).
- **Video Importer → Languages**: Files reference languages only by the numeric Arclight-era `languageId` (English `529` doubles as the "primary subtitle" marker); the Language entities and their audio previews are owned upstream — the importer only attaches the `.aac` preview to an existing Language.
- **Watch → Media (catalog consumption)**: Watch renders the media context's **Video**/**Variant** catalog — a page plays when the Variant has a stream and browses its children otherwise — fetched via the gateway and searched via the Algolia **variant-grained** index (one record per language rendition, unlike the admin's video-grained view). All entity vocabulary is upstream; the app owns the `.html` Watch Path scheme, the Slug Map redirects, and the hand-curated Showcase Rails/campaign pages. Embed codes hand playback off to the Arclight player (`refId` = Variant id).
- **Watch → Languages**: Viewer language preferences (Audio Language, Subtitle Preference) and the site Locale are built on the languages context's catalog, referenced by `languageId` (English `529` default) — the same four-representation trap (id/slug/locale/BCP-47) as Resources. The app caches the language list but owns none of it.
- **Watch ↔ Resources (succession)**: `apps/resources` is the next-generation rebuild of this surface — same `/watch` product and media vocabulary. Both apps still use the Experimental Cookie in their beta-banner UX, but JF Proxy no longer uses it to select a Proxy Destination. Shared terms (Watch Path, Container/Content, Easter Collection Page, the Collection overload) are defined in both glossaries and must be kept aligned.
- **Watch Modern ↔ Watch (proxy coexistence)**: `apps/watch-modern` is a separate successor shell (not a fork) that mounts at the same `/watch` path as the legacy app, with assets namespaced under `/watch/modern` so a proxy can split traffic between them. It carries its own copy of the Legacy Watch Redirect shim (`/bin/jf/watch.html/...`). Terminology trap: three apps answer to "Watch" — legacy `watch`, `resources` (the full rebuild), and this shell; see [Watch Modern CONTEXT.md](./apps/watch-modern/CONTEXT.md).
- **Watch Modern → Media/Languages (intended, not yet consumed)**: The shell consumes no catalog today — no gateway queries, no Algolia. Its configuration points at the gateway and at Cloudflare Stream/Images and WordPress image sources as _intended_ providers, but no video vocabulary is implemented; Video/Variant/Language semantics belong to the sibling surfaces until they land here.
- **Videos Admin → Media (curation)**: The admin is where the media context's catalog is curated — Video, Variant, Edition, Subtitle, Download, Keyword, and Origin are all upstream vocabulary; the admin owns none of them. The admin owns only its display language (an "Audio Language" _is_ a Variant; "Short Description" _is_ the snippet; label `segment` displays as "Clip"), the publishing workflow (Draft, Publishing Validation, Publish All, Locked, Restricted-as-blocklist), and the Variant Upload lifecycle (bytes to R2, encoding handed to Mux — accepted ≠ playable, same deferred-encoding semantics as the Video Importer). Access requires the media-context `publisher` role.
- **Videos Admin → Languages (reference-data editing)**: The Language Admin section edits the languages context's entities directly — localized/native Language names, BCP-47/ISO-3 codes, and CountryLanguage links (speakers, primary/suggested/order) — gated by the languages-context `publisher` role. The `hasVideos` flag is read-only here (set upstream). Caveat: production languages data is dumped downstream (see Languages source of truth), so edits made in lower environments are overwritten on the next import.
- **Videos Admin → Users (authentication)**: Sign-in is Firebase (via the gateway's identity model); the app resolves the caller's `mediaUserRoles`/`languageUserRoles` from the `AuthenticatedUser` federation surface on every request. The only role value is `publisher`, held independently per context; no impersonation exists on this surface.
- **Videos Admin ↔ Algolia (experimental read model)**: The "Algolia (Experimental)" list browses the public Algolia video index (the same index Watch/Resources search) rather than the admin GraphQL list — eventually consistent, read-only, with its own `published` flag distinct from the Video's and the Variant's.
- **Docs → all contexts (description only)**: The docs site describes the other contexts but owns none of their vocabulary; when a docs page and an owning context's `CONTEXT.md` disagree on a term, the glossary wins and the docs page should be corrected. One term is docs-native: **Core Development Kit** (the platform-as-a-whole framing) exists only on this surface.
- **E2E Testing → all app surfaces (verification)**: Each suite exercises one deployed surface from the outside — through the browser for the apps, at the HTTP level for Arclight's REST façade (Contract Specs guarding what external API-key consumers depend on). Specs speak the owning contexts' vocabulary (Journey, Template, Media Component...) but own none of it; authenticated flows share the five-account Playwright User pool, so suites mutate shared state rather than seeded data.
- **Journeys UI → Journeys (rendering the upstream model)**: The kernel renders the Journeys context's Journey/Block/Action entities but owns none of them; its own terms are the client-side forms — **Tree Block** (nested), the **Transformer** (flat → tree), the **Block Renderer** and **Wrappers**. Consumers differ only by the Wrappers they inject.
- **Journeys UI → Journeys Viewer (playback half)**: Block History, Next/Previous Active Block, Action Handling, and the Event Mutations (Journey Analytics + Plausible mirroring) live in the kernel and are exercised only by the viewer; playback position never leaves the device except as events.
- **Journeys UI → Journeys Admin (editing half)**: The Editor Provider (selected step/block, Active Content, Slides, Analytics Mode state) and the undoable **Command** stack live in the kernel and are exercised only by the admin. Terminology trap: a **Slide** here is an editor panel position, never a Step or Card.
- **Journeys UI → Resources / Journeys Admin (gallery components)**: The Template Gallery components and Algolia search providers are shared render surfaces for the Journeys context's Template Gallery; the gallery vocabulary itself is upstream. Trap: the kernel's render mode (`renderMode`: `default`/`admin`/`embed`/`customize`, formerly `variant`) is distinct from the media context's **Variant** — say "render mode".
- **Shared AI → Journeys (AI features)**: `api-journeys`' AI surfaces — journey AI translation, journey-language detection, and simple-journey generation/editing — call LLMs through this kernel's Fallback Sessions and hardened prompts, and speak **Journey Simple** as the model-facing shape. The Journey/Block ↔ Journey Simple mapping (`simplifyJourney`/`updateSimpleJourney`) is owned by the Journeys context; the kernel owns only the schema and its Update-Schema validation gate. Trap: a **Simple Card** is a whole Step (+ its Card), not the Journeys context's `CardBlock`.
- **Yoga → all subgraph APIs (runtime kernel)**: every API builds on this kernel's runtime vocabulary — the **Payload User** (the sanitized projection of the gateway-forwarded Firebase JWT; its `id` is the Firebase UID, not the Users context's record id), the **Interop Token** (shared secret + NAT allowlist — the third trust mechanism alongside the Firebase JWT and the Gateway Signature), the **Gateway Callback Client** (a subgraph calling back into the supergraph as itself), and the **Leader Lock** (electing one ECS instance per Service to run background workers). Boundary leak: journeys/users-shaped email types (`JourneyForEmails`, `UserJourneyRole`, the events-notification job) live in the kernel because its Email Kit renders them; the owning glossaries win on any disagreement.
- **Prisma → API contexts (storage)**: Each API context's Logical Database and Prisma schema live in `libs/prisma/<domain>`; the context owns its models and migrations, the kernel owns only the conventions (client singletons, migrate/seed targets, the analytics introspect-only special case). The analytics schema is _pulled_ from Plausible's externally-owned database — the Prisma face of the Plausible Provisioning context's "introspected external DB" relationship.
- **Prisma cross-context reads (boundary leak)**: The federation story above ("references by id only") holds at the GraphQL surface, but not at the database: api-journeys reads the `users` and `media` databases directly (auth scopes, user-deletion checks, email/cleanup workers) and api-media reads `languages` (hasVideos maintenance, Algolia sync) and `journeys` (journeys access). Any change to those tables is a cross-context breaking change even if the GraphQL contract is untouched.
- **Infrastructure → all Service contexts (hosting)**: The six subgraph APIs and Arclight run as ECS **Services** on the Infrastructure context's Substrate; each owns an **Infrastructure Folder** inside its project dir declaring its Service Config, planned/applied as part of the main Terraform state via Atlantis. Exposure split: `api-gateway` and `arclight` are Public Services; the other APIs are Internal (`<name>.service.internal`). Naming trap: the journeys Service module is named `api-journeys-modern` but is the same `api-journeys` deployable.
- **Infrastructure → frontend surfaces (deploy split)**: The Next.js apps (journeys, journeys-admin, watch, resources, player, short-links, videos-admin, docs...) deploy to **Vercel** via GitHub Actions and Cloudflare Workers handle `workers/*` — neither surface is Terraform-managed. Arclight is dual-surface (ECS and Vercel).
- **Infrastructure → Plausible Provisioning (hosting)**: The Plausible Analytics instance that `api-analytics` introspects runs on the Infrastructure context's **Platform Cluster** (EKS, Argo CD GitOps) — the only EKS workload with product significance; everything else on EKS is ops tooling.
- **Dev Hosts → consuming surfaces (dev-trust gate)**: The kernel answers only "is this hostname a Dev Host?" (fail-closed, exact match, secret-provisioned via Doppler); each consuming surface owns its own Relaxations — Journeys Viewer's proxy rewrite and `allowedDevOrigins`, Journeys Admin's gateway-URL derivation, auth-token handling, and QR-code short-link host swap. Localhost is deliberately out of scope; surfaces carve it out themselves.
- **ESLint → all workspaces (lint policy)**: Every workspace's `eslint.config.mjs` imports exactly one Flavor (or the Common Base directly) by relative path — there is no root ESLint config. The kernel owns the policy vocabulary (Restricted Imports, the Literal-String Rule, the Jest-flavored Test-File Carve-out); workspaces own only their local overrides.
- **GQL → Gateway (schema binding)**: The kernel's Tada Environment is generated from the gateway's checked-in composed `schema.graphql` (the same file the dev gateway writes via `hive dev`); when subgraph schemas change, the composed file and then the Tada Environment must be regenerated or client operation types silently drift. Consumers (arclight, watch, resources, short-links, journeys-admin...) import only the `graphql` tag and `ResultOf`/`VariablesOf` — all operation typing is inference, no per-operation codegen.
- **Shared UI Dynamic → Shared UI (chunk boundary)**: Each Dynamic Entry re-exports exactly one Shared UI component; the façade owns only the webpack chunk boundary, never the component. Consumed today only by journeys-admin's Journey/Template list views via `next/dynamic` (`ssr: false`).
- **Storybook → Registered Projects (catalog aggregation)**: The shell renders other projects' stories but owns none of them; registration is two-sided — a story glob in the Root Config's `storiesForProject` _and_ an entry in the shell's `implicitDependencies` (so `nx affected` rebuilds the catalog when that project changes). Drift trap: the two lists are maintained by hand and can disagree (today `watch` and `videos-admin` have story globs but are not implicit dependencies, so their changes don't retrigger the VR build).
- **Shared UI → frontend surfaces (design kernel)**: The kernel owns the visual vocabulary — Themes are picked per surface (`journeysAdmin` for the admin chrome, `journeyUi` for rendered journey content, `website` for jesusfilm.org styling, `base` underneath) and per journey card; the Icon Set replaces the lint-banned `@mui/icons-material`; Feature Flags are resolved upstream (LaunchDarkly, server-side) and injected into the dumb `FlagsProvider`. The Tailwind-era sibling Shared UI Modern is not part of this kernel.
- **Shared UI Modern → Watch (Tailwind kernel)**: The kernel owns the Vendored Components (shadcn/ui source copied in, never an npm dependency — new ones via the Add Script so Radix deps land at the monorepo root), the HSL-variable Design Tokens (`.dark` class swaps them), and `cn`. Consumed today only by Watch (legacy)'s Tailwind-migrated surfaces (search, dialogs, video controls, collections); Watch Modern and Resources do not use it yet. "Theme" stays the MUI word — here it's Design Tokens.
- **JF Proxy → public surfaces (edge routing)**: The worker owns which sections of `jesusfilm.org` map to which surface — `/watch` goes to the Watch destination, while `/journeys`, `/resources`, and other claimed paths go to the Resources destination. The destinations are opaque per-environment hostnames, so the worker knows surfaces only as hostnames, never as apps.
- **Langfuse Export → Journeys Viewer (observability projection)**: The export tool reads the AI assistant's chat traces from Langfuse (never the app or its database) and re-derives Conversations by session; only Sanitised Conversations leave the pipeline. It writes nothing back — strictly read-only over observability data.
- **Load Test → Journeys Viewer (load generation)**: The load tester drives the viewer's unauthenticated `/api/chat` endpoint on stage; the Vercel Firewall keys on source IP, so Virtual Users model concurrency, not distinct callers. Its runs are what dominates non-production Langfuse traffic (why Langfuse Export defaults to `--environment production`).
- **Scripts → CI/Infrastructure (pipeline glue)**: GitHub Actions consumes Affected Projects/Deployable Services to shape job matrices and the Preview Deployment Comment to report Vercel URLs; the Stage Reset maintains the `stage` branch the Infrastructure context's stage Environment deploys from.
- **Gateway → all subgraphs (composition & trust)**: The gateway composes `api-analytics`, `api-journeys-modern`, `api-languages`, `api-media`, and `api-users` into the supergraph (from the Hive schema registry in stage/prod, a checked-in composed file in dev). It verifies the Firebase JWT once and forwards the decoded payload plus an HMAC Gateway Signature; subgraphs trust forwarded identity only when signed. Exception: the raw `authorization` header passes through untouched to `api-analytics` only (Plausible API key, not Firebase).
