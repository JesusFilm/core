# fix(ci): use a wrapper job so /sync-ce PRs satisfy the required `lint` check

## Overview

PR [#9095](https://github.com/JesusFilm/core/pull/9095) added a job-level `if:` to `autofix.ci.yml` so the `lint` job skips on `/sync-ce`-generated PRs (branches matching `*-chore-ce-sync-*`). The job-level skip works as designed — but a _required_ status check that concludes `SKIPPED` is treated by the `main` ruleset as **not passing**, so `/sync-ce` PRs are now `BLOCKED` from merging instead of breezing through.

This plan replaces the job-level skip with the GitHub-native **wrapper-job pattern**: rename the existing job to `lint-work` (gate intact) and add a thin always-running `lint` wrapper that resolves the required check to `SUCCESS` when `lint-work` either succeeded or was skipped, and `FAILURE` otherwise. Required-check name is preserved, so no ruleset change is needed.

## Problem Statement

Empirical evidence on PR [#9096](https://github.com/JesusFilm/core/pull/9096) (the open `/sync-ce` PR):

- `mergeStateStatus`: `BLOCKED`
- `reviewDecision`: `APPROVED`
- All other status checks: `SUCCESS`
- `lint`: `status=COMPLETED, conclusion=SKIPPED`

GitHub's behaviour here is documented and intentional ([orgs/community#13690](https://github.com/orgs/community/discussions/13690) and the [pull requests with skipped runs](https://docs.github.com/en/repositories/configuring-branches-and-merges-in-your-repository/managing-rulesets/managing-rulesets-for-a-repository#using-merge-queue-with-skipped-status-checks) docs): a job-level `if:` that evaluates to false produces a "skipped" check run. For required status checks, "skipped" is **not** "passed".

PR #9095's stated test plan included "On the next `/sync-ce` PR, confirm the autofix.ci check is skipped (shows as 'Skipped' in the PR checks)" — that part is working correctly. What we missed is that branch-protection treats `SKIPPED` and `PENDING` the same way for required checks.

## Proposed Solution

Apply the wrapper-job pattern to `.github/workflows/autofix.ci.yml`:

1. Rename `jobs.lint` → `jobs.lint-work`, leaving its existing `if:` guard and every step **completely unchanged**.
2. Add a new `jobs.lint` wrapper:
   - `needs: lint-work`
   - `if: always()` (so it runs even when `lint-work` is skipped or fails)
   - `runs-on: ubuntu-latest` (cheapest runner; this is a ~5s job)
   - One `run:` step that exits 0 when `needs.lint-work.result` is `success` or `skipped`, and exits 1 otherwise.

The required check `lint` is now produced by the wrapper. On `/sync-ce` PRs, `lint-work` is skipped, the wrapper exits 0, and the check resolves to `SUCCESS`. On every other event (push to `main`, regular PRs, `merge_group`, `workflow_call`), `lint-work` runs as today; the wrapper passes through its conclusion.

### Concrete diff (target shape)

```yaml
jobs:
  lint-work:
    # Skip /sync-ce PRs so autofix doesn't reformat CE-managed files.
    # The "-chore-ce-sync-" pattern is generated by .claude/skills/sync-ce/SKILL.md.
    if: ${{ !contains(github.head_ref, '-chore-ce-sync-') }}
    runs-on: blacksmith-2vcpu-ubuntu-2204
    strategy:
      matrix:
        node-version: [22]
    env:
      NX_CLOUD_ACCESS_TOKEN: ${{ secrets.NX_CLOUD_ACCESS_TOKEN }}
      NODE_OPTIONS: '--max-old-space-size=8192'
    steps:
      # ... every existing step, untouched ...

  # Required-check shim. Resolves `lint` to SUCCESS when lint-work succeeded
  # OR was skipped (e.g. /sync-ce PRs), and FAILURE otherwise. This avoids
  # the "skipped required check blocks merge" behaviour that bit #9096.
  lint:
    needs: lint-work
    if: always()
    runs-on: ubuntu-latest
    steps:
      - name: Resolve lint-work result
        run: |
          result='${{ needs.lint-work.result }}'
          echo "lint-work concluded: $result"
          case "$result" in
            success|skipped) exit 0 ;;
            *)               exit 1 ;;  # failure, cancelled, etc.
          esac
```

## Technical Considerations

- **Required-check name preservation.** The wrapper's job ID is `lint`, so the check name reported to the ruleset is `lint` — identical to today. No branch-protection / ruleset edit needed.
- **`if: always()` is load-bearing.** Without it, when `lint-work` is skipped the wrapper would itself be implicitly skipped (default `needs:` behaviour for skipped dependencies), reproducing the original bug.
- **`needs.<job>.result` values.** GitHub's documented set is `success | failure | cancelled | skipped`. The case statement defaults `*` → `exit 1`, so any unexpected value fails closed.
- **Cancelled runs fail closed.** If a user cancels a workflow run (only possible for non-sync PRs since sync PRs skip `lint-work`), the wrapper concludes `FAILURE` rather than `SUCCESS`. Right call — a cancelled run shouldn't certify a PR.
- **No secrets, no permissions.** The wrapper reads only `needs.*.result`. `runs-on: ubuntu-latest` (GitHub-hosted) is fine; no need to use the Blacksmith runners for a 5-second job.
- **Cost.** One small ubuntu-latest runner per workflow run, shared across PR/push/merge_group events. Negligible.
- **Two checks now appear instead of one.** PRs will list `lint-work` (success or skipped) AND `lint` (success/failure). Slightly noisier UI; acceptable trade-off.
- **`workflow_call` consumers of this file.** None today — verified by grepping `.github/` for `uses:.*autofix.ci.yml`. If anything ever does call this workflow, only the `lint` job becomes the visible-from-outside check (same as today, same name).
- **Effect on PR #9096 (open /sync-ce PR).** GitHub Actions uses the workflow file from the PR's head commit for `pull_request` events. After this fix merges to `main`, PR #9096 needs a `main` merge into its branch (or the PR re-created) to pick up the wrapper. This is documented in the verification plan below.

## System-Wide Impact

- **Interaction graph.** `pull_request` event → `lint-work` job evaluates `if:` → either runs the full prettier/eslint/type-check pipeline OR is skipped. Either way `lint` wrapper job (with `if: always()`) runs after, reads `needs.lint-work.result`, exits 0/1. Branch-protection sees `lint` conclude SUCCESS/FAILURE. Other workflows (`affected`, `build`, `test`, etc.) unchanged.
- **Error propagation.** Failures in the existing `lint-work` steps still surface in their original logs. The wrapper's only error mode is "exit 1 because lint-work failed", which is observable both via the wrapper's tiny log AND the underlying `lint-work` job's logs. No error swallowing.
- **State lifecycle.** No persistent state. The `autofix-ci/action` step (which auto-pushes lint fixes via a commit) lives inside `lint-work` and is unaffected. It does not run for sync PRs (good — that was the original goal of #9095).
- **API surface parity.** Only one workflow exposes the `lint` check. No other surface to update.
- **Integration test scenarios.** See Verification Plan below — covers both real-PR and sync-PR flows end-to-end.

## Alternative Approaches Considered

The /lfg request explicitly chose Option 2 from a four-option deep dive earlier in the conversation. Recording the rejected alternatives here for posterity:

1. **Step-level `if:` with shared env var.** One source-of-truth value (`env.SHOULD_RUN`), but the `if:` token still has to be repeated on ~15 steps. Future steps added without the gate silently break the skip. Rejected: structurally fragile.
2. **Reusable workflow split.** Move steps into `autofix.ci.work.yml` (callable), have `autofix.ci.yml` become a tiny dispatcher. Cleanest separation if multiple workflows ever need the gate, but over-engineered for one workflow / one job. Hold for the second use case.
3. **Ruleset bypass / drop `lint` from required checks.** Zero workflow code; correct semantic home for "sync-ce PRs are exempt". Rejected: requires admin (Mike) cross-team coordination, weakens the safety net silently if `/sync-ce` ever expands scope, and the bypass condition lives outside git with no PR review trail.

## Success Metrics

- Zero `/sync-ce` PRs `BLOCKED` due to a `SKIPPED` `lint` check after this lands.
- No regression in time-to-green for non-sync PRs (the wrapper adds ~5–10 seconds; not noticeable in the ~5-minute end-to-end CI flow).

## Dependencies & Risks

- **Risk:** A second `lint` check now exists in the PR UI alongside `lint-work`. Mitigated by the wrapper's name being identical to what existed before — anyone scanning for `lint` finds it.
- **Risk:** If `needs.lint-work.result` ever returns a value other than `success | failure | cancelled | skipped`, the wrapper fails closed. Acceptable failure mode.
- **Risk:** PR #9096 will not benefit until it pulls in the new workflow file from `main`. Mitigated by a post-merge step in the verification plan: merge `main` into 9096 and re-check.
- **Dependency on prior commit.** Builds on top of #9095 (the job-level `if:` guard). If #9095 were ever reverted, this change still reads correctly but the `lint-work.if:` wouldn't filter anything, which is a no-op (everything would just always run).
