## Residual Review Findings

- **[P2][gated_auto → downstream-resolver][confidence 50] Public SSR amplifies GraphQL gateway load** — `apps/journeys-admin/pages/templates/index.tsx:108-160` (security reviewer).

  Converting `/templates` from `getStaticProps` (`revalidate: 60`) to `getServerSideProps` removes the implicit Vercel ISR cache. `/templates` is publicly accessible and trivially scriptable; an attacker amplifies ~5x request load against the GraphQL gateway from a single GET. Anchor 50 because actual exposure depends on infra-level rate limits not visible in the diff.

  **Suggested fix:** add `ctx.res.setHeader('Cache-Control', 'public, s-maxage=60, stale-while-revalidate=600')` to mirror the previous caching at the CDN.

  **Why deferred:** adding cache headers to an SSR page that uses cookies for auth has subtle correctness implications. Authenticated and unauthenticated responses differ (the navbar visibility changes, plus `userSerialized` is embedded). Caching authed responses publicly would leak the JWT to other requesters. The fix needs to either:
  - Use `private` directive (no CDN cache, just browser cache — limited benefit), or
  - Add `Vary: Cookie` (CDN partitions by cookie — works but explodes cache key space)
  - Or restructure to render auth-dependent state purely client-side and cache the rest.

  This is a real production concern but warrants explicit human design rather than an automatic application.
