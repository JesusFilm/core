# Pothos `_query` Parameter Ignored — Empty Manage Editors Dialog

## Problem

The "Manage Editors" dialog (`AccessDialog`) showed an empty editor list for all users (owners, editors, team managers, team members). No error was visible in the UI. The browser console showed a CORS error (unrelated) and the GraphQL response contained both valid data AND an error — but the UI rendered nothing.

## Symptoms

- "Manage Editors" dialog opens but the section where editors should appear is blank
- No error message shown to the user
- GraphQL `GetJourneyWithPermissions` query returns both `data` and `errors` in the response
- Error at path `journey.team.userTeams[0].journeyNotification`: `Cannot read properties of undefined (reading '0')`
- Bug only manifests for **journeys that have a team** — teamless journeys may work because `UserJourney.journeyNotification` uses `t.relation` (has fallback)

## Investigation Steps

1. User reported empty "Manage Editors" dialog with CORS errors in console.
2. Fetched Linear tickets NES-1480 and NES-1481 for context.
3. Traced `AccessDialog` component to the `GET_JOURNEY_WITH_PERMISSIONS` query targeting `adminJourney`.
4. Found `adminJourney.query.ts` line 22: `resolve: async (_query, ...)` — the underscore prefix signals intentional non-use.
5. Found `UserTeam.journeyNotification` uses Pothos `select` directive (requires `_query` propagation) while `UserJourney.journeyNotification` uses `t.relation` (has automatic fallback).
6. Confirmed via Pothos docs: `select` fields have NO fallback when `_query` is ignored; `t.relation` fields DO have fallback.
7. User confirmed GraphQL response contains both data and error — Apollo Client's default `errorPolicy: 'none'` discards all data when errors present.
8. Also discovered `Query.adminJourney` missing from response cache TTL config.

## Root Cause

The `adminJourney` resolver in `adminJourney.query.ts` ignored the Pothos `_query` parameter (prefixed with underscore, never spread into the Prisma call). The `_query` parameter carries nested `select`/`include` directives generated by Pothos based on the GraphQL selection set. When ignored:

1. `UserTeam.journeyNotification` (uses `select` directive) expects `journeyNotifications` to be loaded on the parent object
2. Since `_query` was skipped, `journeyNotifications` was never included in the Prisma query
3. The field resolver accesses `journeyNotifications[0]` on `undefined` → `TypeError`
4. GraphQL catches the error, nullifies the field, adds error to response
5. Apollo Client (default `errorPolicy: 'none'`) sees `errors` array → discards ALL valid `data`
6. `AccessDialog` receives `data === undefined` → renders empty lists

**Key insight — `select` vs `t.relation` fallback behavior:**
- **`select` on a field:** Tells Pothos to add data to the parent's `query`. If parent ignores `query`, data is simply **missing** — no fallback, no separate query. Crashes.
- **`t.relation`:** Has automatic fallback — Pothos issues efficient `findUnique` queries to resolve missing data, batched for performance. Works even if `query` is ignored.

This is why `UserTeam.journeyNotification` (select) crashed while `UserJourney.journeyNotification` (t.relation) worked fine.

## Solution

### Two-query pattern in `adminJourney.query.ts`

```typescript
resolve: async (query, _parent, args, context) => {
  const where: Prisma.JourneyWhereUniqueInput =
    args.idType === 'slug'
      ? { slug: String(args.id) }
      : { id: String(args.id) }

  // Fetch with ACL-required includes for authorization check.
  // This is intentionally a separate query from the data fetch below
  // because Pothos query may contain select directives that would
  // strip the relations needed for journeyAcl().
  const journeyForAcl = await prisma.journey.findUnique({
    where,
    include: {
      userJourneys: true,
      team: { include: { userTeams: true } }
    }
  })

  if (journeyForAcl == null)
    throw new GraphQLError('journey not found', {
      extensions: { code: 'NOT_FOUND' }
    })

  if (!journeyAcl(Action.Read, journeyForAcl, context.user))
    throw new GraphQLError('user is not allowed to view journey', {
      extensions: { code: 'FORBIDDEN' }
    })

  // Fetch with Pothos query for correct nested field resolution
  // (select directives like UserTeam.journeyNotification require this)
  return prisma.journey.findUniqueOrThrow({ ...query, where })
}
```

**Why two queries instead of merging?** Pothos `query` may contain `select` directives (not just `include`). Prisma does not allow both `select` and `include` at the same level. Deep-merging is fragile. The two-query approach cleanly separates ACL concerns from data fetching.

### Response cache fix in `yoga.ts`

Added `'Query.adminJourney': 0` to `ttlPerSchemaCoordinate`. With `session: () => null` (shared cache across all users), any authenticated query without TTL 0 risks serving one user's ACL-gated data to another user.

## Prevention

### Immediate (highest leverage)

1. **Set `onUnusedQuery: 'error'` in dev/test** — One-line change in the Pothos builder config. Currently set to `'warn'` which only logs. Changing to `'error'` would have caught this bug immediately in any test that exercised the resolver.

2. **Ban `_query` naming in `prismaField` resolvers** — The underscore prefix convention signals "intentionally unused," which is always wrong for `prismaField`. A CI grep check (`rg '_query' --glob '*.ts' | xargs rg 'prismaField'`) catches this pattern.

3. **Set Apollo Client `errorPolicy: 'all'` as default** — Prevents silent data loss across all queries. Partial data is returned to the UI, and errors are surfaced in the `error` field.

### Test patterns

- Always query at least one **nested selected sub-field** in `prismaField` integration tests
- Always assert `result.errors` is undefined/empty — many tests only check `data`, which passes even when errors exist
- For new `prismaField` resolvers, verify `findUniqueOrThrow` receives the Pothos query parameter via `expect.objectContaining`

### Observability

- Add an Apollo Link that logs when `response.errors` is non-empty with `errorPolicy: 'none'`
- Alert on error rate spikes per GraphQL operation on the server side
